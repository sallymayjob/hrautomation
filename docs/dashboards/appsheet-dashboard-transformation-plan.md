# AppSheet Dashboard Transformation Plan

**Documentation status:** Refreshed for Apps Script-native operations baseline (configuration, security, trigger reconciliation, and governed workflows). Canonical deployment/run sequence lives in `DEPLOYMENT.md`.


## 1. Executive Summary
The current system already has a solid operational-dashboard foundation that maps well to AppSheet: onboarding + checklist workflows, training due/overdue tracking, governed approvals, and weekly summary sheets generated by Apps Script. Existing dashboard specs define KPI cards and pivot dimensions for onboarding, training, and audit domains, and weekly summary tabs are already materialized in Sheets by `Reporting.gs`.

Recommended approach:
- Move **operational dashboards** (pending approvals, overdue queues, blocked onboarding, team-owner worklists, SLA-at-risk lists) into AppSheet.
- Keep **heavy historical/executive analytics** outside AppSheet (Looker Studio/BI), fed from stable summary extracts.
- Reuse existing Apps Script summary generation for expensive aggregates, and add a minimal number of helper columns/tabs for AppSheet UX.

This is a practical migration (not a warehouse redesign): preserve operational usefulness, keep formulas simple, and keep write governance in existing Apps Script pathways.

## 2. Current Dashboard Components Found in the Codebase
### Existing dashboard specs and KPI definitions
1. **Onboarding dashboard spec** (`Onboarding_Dashboard`, `Onboarding_KPI`, `Onboarding_Pivot`) with KPI cards: active, completed, blocked, checklist completion rate, overdue task count.
2. **Training dashboard spec** (`Training_Dashboard`, `Training_KPI`, `Training_Pivot`) with KPI cards: assigned, completed, in-progress, overdue, on-time completion rate.
3. **Audit dashboard spec** (`Audit_Dashboard`, `Audit_KPI`, `Audit_Pivot`) with KPI cards: total events, last 24h, unique actors, status-change events, entity change rate.

### Script-generated reporting logic
`gas/Reporting.gs` already computes and writes weekly operational summaries into Sheets:
- `Summary - By Employee`
- `Summary - By Team Owner`
- `Summary - By Category`
- `Summary - Blocked Onboarding`

It computes:
- training totals/completions/overdues/due-in-7-days,
- onboarding task totals/done/overdue/completion %,
- unresolved task rollups,
- blocked onboarding prioritization.

### Workflow-linked operational queues already present
- Due/overdue reminder logic in `gas/Reminders.gs` (training reminders + escalation after overdue threshold).
- Status-driven onboarding lifecycle and blocked-state handling in `gas/OnboardingController.gs`.
- Approval + submission state model in `approvals` and `submissions` schemas (`approval_status`, submitted/approved timestamps, actor fields).
- Read-only Slack status commands confirm that Sheets are system-of-record for status edits (`gas/Commands.gs`).

## 3. What Should Move to AppSheet
### A) Onboarding Operations Dashboard (strong fit)
Replace current sheet dashboard tabs with AppSheet dashboard composed of:
- KPI cards (virtual columns or small precomputed KPI table)
- table/deck views for active onboarding records
- inline child checklist list per onboarding detail
- blocked queue and overdue checklist queue slices

Data source:
- `Onboarding` + `Checklist` tables
- optionally `Summary - By Employee` for precomputed completion/overdue counts

User interaction:
- filter by role/team/manager
- drill-down: KPI -> queue list -> onboarding detail -> child checklist tasks
- actions: mark task status (if allowed), trigger reminder/escalation action (bot webhook/AppSheet action)

### B) Training Queue Dashboard (strong fit)
AppSheet views:
- overdue training queue
- due-soon queue (0-7 days)
- owner workload list
- completion status chart by module/owner

Data source:
- `Training` table
- optional precomputed `Training_KPI` helper table

Drill-down:
- chart/list -> training assignment detail -> employee onboarding context (via employee/onboarding reference)

### C) Approvals Workbench Dashboard (strong fit)
AppSheet tables/slices:
- `submissions` (pending/approved/rejected)
- `approvals` (audit trail)

Views:
- pending approvals queue
- my-approvals queue
- decision history

Actions:
- approve/reject actions that update status (if governance allows AppSheet writes), or launch controlled Apps Script endpoint for governed mutation.

### D) Team Operations Dashboard (strong fit)
Use existing weekly summary sheets as lightweight AppSheet sources:
- `Summary - By Team Owner`
- `Summary - By Category`
- `Summary - Blocked Onboarding`

This gives fast dashboard cards/charts without expensive per-row recomputation in AppSheet.

## 4. What Should Stay Outside AppSheet
Keep outside AppSheet (BI/reporting layer):
1. Long-range trend analysis across large historical windows (month/quarter/year over many dimensions).
2. Complex executive scorecards requiring blended cross-source analytics and advanced visual design.
3. Heavier compliance analytics (entity change-rate trend decomposition, anomaly analytics) beyond operational monitoring.

Practical split:
- AppSheet = daily operational workflow control tower.
- BI tool = executive/performance analytics and deep history.

## 5. Required Data Model Changes
Smallest safe changes for AppSheet readiness:

1. **Add explicit AppSheet keys if missing in source tabs**
   - Onboarding key: `onboarding_id` (already present).
   - Checklist key: `task_id` (already present).
   - Training key: add `training_id` synthetic key if currently relying only on `employee_id + module_code`.
   - Submissions/Approvals keys already present (`submission_id`, `approval_id`).

2. **Normalize status enums for AppSheet slices**
   - Onboarding status currently includes values like `PENDING`, `IN_PROGRESS`, `BLOCKED`, `COMPLETE`.
   - Checklist status currently mix (`DONE`/`COMPLETE`) in logic; normalize to one canonical set in sheet + automation mapping.
   - Training status values should stay canonical (`Not Started`, `In Progress`, `Completed`, `Overdue`) with uppercase helper VC for slice predicates.

3. **Strengthen relationships (Refs)**
   - Checklist `[onboarding_id]` -> Onboarding.
   - Training `[employee_id]` -> Onboarding `[onboarding_id]` if semantically same ID; if not, add explicit `onboarding_id` in training for reliable drill-down.
   - Submissions/Approvals -> target entities via `entity_key` (plus optional resolved ref columns).

4. **Add helper SLA fields (precomputed preferred)**
   - `days_until_due`
   - `is_overdue`
   - `sla_risk_band` (e.g., `ON_TRACK`, `DUE_SOON`, `OVERDUE`, `BREACH_RISK`)

Compute via Apps Script or sheet formulas to avoid repeated AppSheet expression cost on large tables.

5. **Materialize lightweight operational summaries**
   - Keep using weekly summary generation and add daily refresh if needed for near-real-time app dashboards.
   - Add helper tabs for role KPI cards if AppSheet performance degrades with virtual aggregate expressions.

## 6. AppSheet Tables, Slices, and Views Plan
### Core tables
- `Onboarding`
- `Checklist`
- `Training`
- `submissions`
- `approvals`
- `Audit` (read-only in app)
- Summary helper tables (`Summary - By Employee`, `Summary - By Team Owner`, `Summary - By Category`, `Summary - Blocked Onboarding`)

### High-value slices
1. `Onboarding_Active`: status in PENDING/IN_PROGRESS/BLOCKED.
2. `Onboarding_Blocked`: status = BLOCKED.
3. `Checklist_Overdue_Open`: not complete and due date < today.
4. `Checklist_Due_3_Days`: open tasks due in <=3 days.
5. `Training_Overdue`: training_status = Overdue OR due date passed and not completed.
6. `Training_Due_This_Week`: due in 0-7 days and not completed.
7. `Approvals_Pending`: approval_status = PENDING.
8. `My_Approvals_Pending`: pending + approver email = USEREMAIL().
9. `My_Team_Worklist`: owner_team or manager_email scoped to current user role.

### Virtual columns (minimal set)
- `[DaysUntilDue]`
- `[IsOverdue]`
- `[CompletionPct]` (for onboarding via related checklist child rows)
- `[OpenTaskCount]`
- `[OverdueTaskCount]`
- `[SlaRisk]`

### View plan
- **Table views**: queues (pending approvals, overdue tasks, blocked onboarding).
- **Deck views**: onboarding cards for managers.
- **Detail views**: onboarding detail with inline related checklist and training rows.
- **Chart views**: status distribution and owner/team workload.
- **Calendar view**: due dates for checklist/training tasks.
- **Dashboard views**: role-specific composites (KPIs + queue + chart).

### Drill-down pattern (standard)
KPI card -> queue table slice -> record detail -> related child records -> action buttons.

## 7. Role-Based Dashboard Design
### Admin
- KPIs: total active workflows, blocked count, overdue count, pending approvals, automation exceptions.
- Data scope: all records.
- Actions: override/resolve blocked items, trigger refresh jobs, assign owners.
- AppSheet: unrestricted security filter, admin dashboard view with exception queues.

### HR
- KPIs: active onboarding, checklist completion rate, missing docs, reminders due.
- Data scope: HR-owned phases + all onboarding core data.
- Actions: update onboarding status fields, assign/retarget tasks, send reminders.
- AppSheet: HR slices by owner_team/status, HR dashboard + onboarding detail drill-down.

### Manager
- KPIs: team onboarding progress, team overdue tasks, training completion %. 
- Data scope: rows where `manager_email = USEREMAIL()`.
- Actions: acknowledge escalations, mark manager-owned checklist tasks, follow-up reminders.
- AppSheet: manager security filter + “My Team” dashboard.

### Approver
- KPIs: pending approvals count, aging approvals, rejected/clarification needed.
- Data scope: approvals assigned to approver or role group.
- Actions: approve/reject/comment.
- AppSheet: approvals workbench dashboard; action buttons bound to governed process.

### Operations Staff
- KPIs: queue length by owner_team/phase, due-today tasks, overdue >3 days.
- Data scope: owner_team-aligned rows.
- Actions: bulk progress updates, exception tagging, note updates.
- AppSheet: team queue dashboard + quick-edit task tables.

### Employee/User (if enabled)
- KPIs: personal onboarding checklist progress, personal training due/overdue.
- Data scope: own onboarding/training rows only.
- Actions: update completion evidence where policy allows.
- AppSheet: minimal personal dashboard and detail-only views.

## 8. KPI Design: AppSheet vs Backend vs BI
### AppSheet operational KPIs (compute in app or small helper table)
- pending approvals count
- blocked onboarding count
- overdue open checklist count
- due this week training count
- my-team queue counts
- personal completion %

### Backend-precomputed KPIs (Sheets formulas/Apps Script jobs)
- onboarding completion % by employee/team/phase
- top unresolved tasks text rollups
- weekly digest totals (already in `Reporting.gs`)
- owner/team/category completion rates
- SLA breach-risk buckets when involving multi-step logic

### BI-only KPIs
- multi-quarter trend analysis
- advanced cohort analytics
- high-cardinality cross-domain metrics requiring heavy joins
- executive narrative dashboards with custom visualization requirements

Computation ownership guidance:
- AppSheet expression: simple row-level booleans/counts and short aggregations.
- Sheets formulas: lightweight per-row transforms and stable helper columns.
- Apps Script summary jobs: expensive aggregations and text rollups.
- External BI: long-horizon and advanced analytics.

## 9. Workflow-Linked Dashboard Recommendations
Surface the following directly in AppSheet:

1. **Pending approvals**
   - Slice from `submissions`/`approvals` where `approval_status = PENDING`.
   - Show aging indicator from `submitted_at`.

2. **Overdue actions**
   - Checklist/training slices using due date + status.
   - Format rules: amber (due soon), red (overdue), dark red (escalation threshold).

3. **SLA breach risk**
   - Use helper `[SlaRisk]` column and chart by risk band.

4. **Onboarding stage counts**
   - Status chart from Onboarding table (`PENDING`, `IN_PROGRESS`, `BLOCKED`, `COMPLETE`).

5. **Completion rates**
   - Employee/team completion from precomputed summary tabs.

6. **Missing documents / exceptions**
   - Slice using `blocked_reason` not blank or required checklist task not complete.

7. **Reminders due**
   - Queue for items matching reminder thresholds (3 days, due today, overdue), aligned with existing reminder logic.

8. **Task ownership views**
   - Owner_team and manager_email slices for team-level operations.

9. **Attendance/training status**
   - Use training status queue and due-date calendar.

## 10. Migration Steps
1. **Inventory and freeze current dashboard tabs**
   - Keep current Sheets dashboard tabs as baseline during transition.

2. **Prepare AppSheet-ready data contracts**
   - Confirm keys/status enums/refs; add helper fields (`days_until_due`, `is_overdue`, `sla_risk_band`).

3. **Stand up core AppSheet tables + relationships**
   - Onboarding, Checklist, Training, Submissions, Approvals, Summary tabs.

4. **Build v1 operational dashboard (recommended first: HR Onboarding Ops)**
   - KPI strip + blocked queue + overdue checklist queue + onboarding detail drill-down.

5. **Add role-based slices and security filters**
   - Manager, approver, operations, admin experiences.

6. **Map actions to governance model**
   - Keep governed commits in Apps Script where required; AppSheet actions should trigger approved pathways.

7. **Performance hardening**
   - Shift expensive virtual aggregations to summary tabs if latency appears.

8. **Parallel run and acceptance**
   - Compare AppSheet dashboard outputs with existing weekly summary tabs for 1-2 cycles.

9. **Retire redundant Sheet dashboard tabs**
   - Keep only helper/precomputed tabs needed by AppSheet and BI export.

## 11. Risks and Caveats
1. **Status vocabulary drift** (`DONE` vs `COMPLETE`, mixed case training values) can break slices.
2. **Weak training-to-onboarding relationship** if training uses `employee_id` without explicit onboarding ref.
3. **Large virtual column computation** can degrade app performance; precompute where possible.
4. **Governed write constraints** mean some AppSheet “edit” actions should call controlled backend flows, not direct writes.
5. **Audit duality confusion** (`Audit` canonical vs training `Audit_Log` telemetry) must be explicit in app table naming.

## 12. Recommended First Dashboard to Build
**HR Onboarding Operations Dashboard (v1)**

Why first:
- Highest operational impact.
- Uses existing strong data model (`Onboarding` + `Checklist`).
- Directly tied to workflow state and reminders.
- Clear drill-down/action loops for internal teams.

v1 components:
- KPI cards: active onboarding, blocked onboarding, overdue tasks, checklist completion rate.
- Queues: blocked onboarding, overdue checklist by owner team, due today.
- Detail drill-down: onboarding record with inline child checklist and top unresolved tasks.
